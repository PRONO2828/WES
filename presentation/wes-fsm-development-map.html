End-to-End Workflow Map For Developing WES Field Service Management

This page shows how dispatch, field execution, workshop follow-up, review, and report release move from left to right. It is grounded in the current WES FSM workspace flow and the API rules that drive status changes.

Primary roles Admin, Technician, Workshop Technician, Review Desk
Main entities Users, Jobs, Sites, Field Reports, Workshop Reports, Forum Messages
Output Approved PDF report plus permanent repeat-site history

Admin Jobs

Dispatch And Assignment

2
Create job

POST /jobs creates a new job in Pending and assigns it to a field technician.

Context follows the work

GET /jobs/:jobId returns job data, site details, report history, and workshop follow-up.

Side channel stays open

Forum messages and presence updates support handoff notes before the technician starts work.

->
Technician Workshop

Field And Workshop Execution

3
Start job

POST /jobs/:jobId/start moves the job from Pending to In Progress.

Capture field evidence

Technician records diagnosis, work done, GPS, measurements, materials, before and after photos, and client signature.

Optional workshop branch

POST /jobs/:jobId/workshop-reports logs bench repair details, parts, costs, and workshop photos.

->
Technician Review Queue

Submission And Review

4
Submit field report

POST /jobs/:jobId/reports creates the report as Pending review and moves the job to Completed.

Office review

GET /review-queue lets admin review details, site history, workshop follow-up, and evidence.

Decision point

POST /reports/:reportId/review sets the review result and controls the next job state.

->
Decision Output

Release, Feedback, And Reuse

5
Approved path

Job becomes Approved, the report becomes exportable, and GET /reports/:reportId/export produces the PDF.

Rejected path

Job returns to In Progress, the technician revisits the site, and new evidence re-enters review.

Knowledge retention

Approved reports stay attached to the site, so future repeat visits start with historical context.

Cross-cutting coordination

Forum messages, SSE updates, and presence tracking support each stage without replacing the core job and report flow.

Rejection loop

Rejected report -> job goes back to In Progress -> technician revisits -> evidence is resubmitted.

Approval loop

Approved report -> PDF export -> site history becomes richer -> next repeat-site dispatch starts smarter.

WES FSM Development Map

Architecture Flow From UI To API To Data To Output

Use this page as the developer reference. It shows which screens talk to which routes, which rules mutate state, and where reusable site knowledge is stored and returned.

Backend service Express app mounted at /api with auth, jobs, reports, search, users, and forum routes
Current data engine In-memory seeded store from createStore() with EventEmitter for forum and presence
1. User-facing surfaces Screens
Login Dashboard Jobs Field Reports Approvals Workshop Reports Center Global Search Forum

Admin gets dispatch and approval views. Technicians get assigned jobs, capture, workshop, and their own report-center history.

v
2. Client state and workflow logic UI orchestration
Session Active View Selected Job Selected Report Uploads Filters Theme Navigation History

Client logic decides what the current user may see, pins repeat-site history into context panels, and prepares payloads for jobs, reports, review, and workshop follow-up.

v
3. API contract Transport layer
/auth/login /dashboard /jobs /jobs/:jobId/start /jobs/:jobId/reports /jobs/:jobId/workshop-reports /reports /review-queue /reports/:reportId/review /reports/:reportId/export /sites /search /forum/messages /forum/stream /users

These routes are the stable handoff points between the UI and the domain logic, and they are where future mobile or React clients should connect.

v
4. Domain rules and state transitions Business logic
Role authorization Job visibility Repeat-site mapping Evidence validation Review decisions PDF export gate Forum events Presence updates

The store enforces the important rules: only assigned technicians can start or report on jobs, evidence is mandatory before submission, and only approved reports may export to PDF.

v
5. Persistence, eventing, and outputs Knowledge layer
Users Jobs Sites Field Reports Workshop Reports Activity Logs Forum Messages EventEmitter PDFKit Export

Everything ends here: approved reports become PDFs, site history becomes richer, and the same stored data feeds later job context, review context, search results, and dashboards.

Recommended Development Order

If the team is actively building or refactoring WES FSM, this order reduces rework by establishing the job and site model first, then layering capture, review, collaboration, and hardening on top.

Dependency path: Foundation -> Dispatch Core -> Field Capture -> Review And Release.

Search And Collaboration should read from the existing job and report models instead of inventing parallel data paths.
Final hardening after these stages: persistent database, object storage for images, JWT secret management, audit logging, mobile packaging, and deployment.
